# SMPTE ST 12-1:2014 — Time and Control Code (Linear Timecode)

_Source: SMPTE ST 12-1:2014 "Time and Control Code", fetched directly from
`https://pub.smpte.org/pub/st12-1/st0012-1-2014.pdf` (41 pp.), render-verified
page-by-page for every bit-position/value table below (`pdftotext -layout`
output cross-checked against the rendered PDF page image for Tables 1–5)._

This document curates the **§9 Linear Time Code (LTC) codeword** — the only
part of ST 12-1 this crate implements. ST 12-1 also defines §10 Vertical
Interval Time Code (VITC), §11 the LTC/VITC relationship, and §12 higher
(48/50/60fps) progressive frame-rate variants; those are out of scope for this
crate (see "Scope" below).

## Scope

This crate models the **logical 80-bit LTC codeword** only — the BCD time
address, flag bits, binary groups ("user bits"), and synchronization word, as
an already-demodulated sequence of bits/bytes. It does **not** implement the
§9.3 biphase-mark-encoded physical/analog audio waveform that carries LTC on
a wire (that is a line-encoding/signal-processing layer, analogous to this
project never decoding PCM/AC-3 audio samples or ANC biphase framing) — the
crate's `Parse`/`Serialize` operate on a 10-byte (80-bit) slice that is
assumed to already be demodulated into bits.

## §8 Structure of the Time Address and Control Bits

### §8.1 Numeric Code

> The numeric code consists of sixteen groups, eight four-bit groups
> containing time address and flag bits, and eight four-bit binary groups for
> user-defined data and control codes.

### §8.2 Time Address

> The basic structure of the time address is based upon the BCD system,
> using units and tens in digit pairs for hours, minutes, seconds, and
> frames. Some of the digits are limited to values that do not require all
> four bits to be significant. These bits shall be omitted from the time
> address and include the "80's" and "40's" of hours, "80's" of minutes,
> "80's" of seconds, and the "80's" and "40's" of frames. Thus the entire
> time address is coded into 26 bits.

Valid ranges (§5.2/§6.2/§7.2, "24-hour clock beginning with 00 hours, 00
minutes, and 00 seconds to 23 hours, 59 minutes, and 59 seconds"):

- hours: `00`–`23`
- minutes / seconds: `00`–`59`
- frames: `00`–`29` (30-frame, non-drop or NTSC drop-frame counting, §5.2.1/
  §5.2.2), `00`–`24` (25-frame, §6.2 — "There is no counting mode such as
  drop frame ... applicable to 25-frame counting"), `00`–`23` (24-frame,
  §7.2)

This crate's typed constructor validates the universal BCD-digit-width bounds
(hours `0..=23`, minutes/seconds `0..=59`, frames `0..=29` — the widest of the
three per-rate frame ranges above, since the 80-bit codeword itself carries no
self-describing frame-rate field to pick a narrower bound against). A caller
that knows its stream's frame rate is responsible for enforcing the tighter
`00`–`24` / `00`–`23` bound itself.

### §8.3 Flag Bits — §8.3.1 Drop frame flag

> This flag shall be set to logical one when drop-frame compensation is
> being performed as specified in Section 5.2.2. When the count is not
> drop-frame compensated, this flag bit shall be set to logical zero.

### §8.3.2 Color frame flag (NTSC and PAL composite color systems)

> If this flag is set to logical one it shall indicate that color frame
> identification has been applied to the time address as defined in Section
> 5.3 or Section 6.3. If this flag is set to logical zero then there is no
> implied relationship between the color frame sequence and the time
> address.

### §8.3.3 Binary group flags — Table 1 (p. 13, render-verified)

> Three flags provide eight unique combinations which signify the use of the
> binary groups (see Section 8.4). Three combinations of these flags also
> specify that the time address count is referenced to a time-of-day clock
> reference (see Section 8.5) ... Note: The binary groups are commonly
> referred to as "user bits".

**Table 1 – Binary group flag assignments**

| BGF2 | BGF1 | BGF0 | Time address reference | Binary group        | Reference section          |
|------|------|------|-------------------------|----------------------|-----------------------------|
| 0    | 0    | 0    | Unspecified             | Unspecified          | §8.4.1                      |
| 0    | 0    | 1    | Unspecified             | 8-bit codes          | §8.4.2                      |
| 0    | 1    | 0    | Clock time              | Unspecified          | §8.4.3, §8.5                |
| 0    | 1    | 1    | Reserved                | Reserved             | §8.4.4                      |
| 1    | 0    | 0    | Unspecified             | Date and time zone   | §8.4.5                      |
| 1    | 0    | 1    | Unspecified             | Page/line            | §8.4.6                      |
| 1    | 1    | 0    | Clock time              | Date and time zone   | §8.4.7, §8.5                |
| 1    | 1    | 1    | Clock time              | Page/line            | §8.4.8, §8.5                |

### §8.3.4 Modulation method specific flag

> The remaining flag bit is reserved for use by each modulation method. This
> flag is defined in Section 9.2.5 and Section 10.2.4.

(§9.2.3 identifies this as the LTC **biphase mark polarity correction** bit.)

## §9 Linear Time Code Application

### §9.1 Codeword Format

> Each LTC codeword consists of 80 bits numbered 0 through 79. The bits are
> generated serially beginning with bit 0. Bit 79 of the codeword is followed
> by bit 0 of the next codeword.

### §9.2.1 Time address — Table 2 (p. 15, render-verified)

> The lowest numbered bit of each group corresponds to the least significant
> bit of each BCD digit.

**Table 2 – LTC time address bit positions**

| Bit    | Definition          |
|--------|---------------------|
| 0–3    | Units of frames     |
| 8–9    | Tens of frames      |
| 16–19  | Units of seconds    |
| 24–26  | Tens of seconds     |
| 32–35  | Units of minutes    |
| 40–42  | Tens of minutes     |
| 48–51  | Units of hours      |
| 56–57  | Tens of hours       |

### §9.2.2 Flag bits — Table 3 (p. 15, render-verified)

> Note that not all flag bits are used by all systems, as designated by the
> symbol "–". Unused flag bits shall be set to 0 by original sources and
> ignored by receiving equipment.

**Table 3 – LTC flag bit positions**

| 30-frame bit | 25-frame bit | 24-frame bit | Definition               |
|--------------|--------------|--------------|---------------------------|
| 10           | – [10]       | – [10]       | Drop frame flag           |
| 11           | 11           | – [11]       | Color frame flag          |
| 27           | 59           | 27           | Polarity correction       |
| 43           | 27           | 43           | Binary group flag BGF0    |
| 58           | 58           | 58           | Binary group flag BGF1    |
| 59           | 43           | 59           | Binary group flag BGF2    |

**Judgment call (important, spec-verified quirk, not a transcription typo):**
the drop-frame flag (bit 10) and color-frame flag (bit 11) sit at **fixed**
bit positions across all three frame rates — a system that doesn't use the
concept (e.g. drop-frame at 25/24-frame) just leaves the bit reserved-zero at
that position. But the **polarity-correction / BGF0 / BGF1 / BGF2 bits move**:
30-frame and 24-frame share one mapping (27→polarity, 43→BGF0, 58→BGF1,
59→BGF2), while **25-frame swaps bit 27 and bit 59's meaning**
(59→polarity, 27→BGF0, 58→BGF1, 43→BGF2). This was verified against the
rendered PDF table image (not just the text-layer extraction) because a
column-transposition bug in extraction would silently produce a plausible
but wrong table. Since the 80-bit codeword carries no self-describing
frame-rate field, resolving these four bits' meaning requires the *caller* to
supply the frame rate — see [`FrameRate`](../src/frame.rs) and
`LtcFrame::polarity_correction`/`LtcFrame::binary_group_flags`.

### §9.2.3 Biphase mark polarity correction

> Because of the nature of the modulation method, the polarity of the first
> clock transition of the first bit of the synchronization word may differ
> from codeword to codeword depending on the number of logical zeros in the
> data. ... When polarity correction of the codeword is required and the
> number of logical zeros in bit positions 0 through 63 (exclusive of the
> polarity correction bit itself) is odd, then the polarity correction bit
> shall be set to logical one, otherwise the polarity correction bit shall be
> set to logical zero.

This bit is specific to the §9.3 biphase-mark modulation layer, which is out
of this crate's scope (see "Scope" above); the crate exposes its raw value
(resolved per `FrameRate`) but does not compute or validate it against the
zero-count rule, since that rule is about the physical modulation, not the
logical codeword content.

### §9.2.4 Binary groups — Table 4 (p. 16, render-verified)

> Eight four-bit binary groups are defined in Section 8.3. The lowest
> numbered bit of each group corresponds to the least significant bit of that
> group.

**Table 4 – LTC binary group bit positions**

| Bit    | Definition          |
|--------|---------------------|
| 4–7    | First binary group  |
| 12–15  | Second binary group |
| 20–23  | Third binary group  |
| 28–31  | Fourth binary group |
| 36–39  | Fifth binary group  |
| 44–47  | Sixth binary group  |
| 52–55  | Seventh binary group|
| 60–63  | Eighth binary group |

### §9.2.5 Synchronization word — Table 5 (p. 16, render-verified)

> The synchronization word is a static combination of bits which can be used
> by receiving equipment to accurately identify the bit position of the
> serial code. ... Bits 65 to 78 form a unique pattern that is symmetrical
> about the center of the synchronization word, allowing detection in either
> direction. Bits 64 and 79 are complements of each other, allowing a
> receiver to determine the direction of the code.

**Table 5 – LTC synchronization word bit positions and values**

| Bit | Value | Bit | Value |
|-----|-------|-----|-------|
| 64  | 0     | 72  | 1     |
| 65  | 0     | 73  | 1     |
| 66  | 1     | 74  | 1     |
| 67  | 1     | 75  | 1     |
| 68  | 1     | 76  | 1     |
| 69  | 1     | 77  | 1     |
| 70  | 1     | 78  | 0     |
| 71  | 1     | 79  | 1     |

## Byte packing convention (this crate's choice, not spec-mandated wording)

ST 12-1 numbers bits 0–79 and defines fields purely in terms of that bit
index; it does not itself say "byte 0, byte 1, ...". This crate packs bit `N`
into `bytes[N / 8]`, bit position `N % 8` within that byte (bit 0 of a byte =
its LSB) — the universally-used convention for LTC (every byte-oriented LTC
reader/writer in practice, and the reason the well-known LTC sync-word byte
pair is `0xFC 0xBF`, matches this packing exactly; see the worked vector
below). Under this convention every field in Tables 2–5 falls **within a
single byte** (no field straddles a byte boundary), which is what makes the
packing unambiguous:

| Byte | Bits  | Low nibble (bits 0–3 of byte) | High nibble (bits 4–7 of byte)     |
|------|-------|--------------------------------|--------------------------------------|
| 0    | 0–7   | frame units                    | binary group 1                       |
| 1    | 8–15  | frame tens (2b) + drop(1) + color(1) | binary group 2                |
| 2    | 16–23 | seconds units                  | binary group 3                       |
| 3    | 24–31 | seconds tens (3b) + flag bit 27 | binary group 4                      |
| 4    | 32–39 | minute units                    | binary group 5                      |
| 5    | 40–47 | minute tens (3b) + flag bit 43  | binary group 6                      |
| 6    | 48–55 | hour units                      | binary group 7                      |
| 7    | 56–63 | hour tens (2b) + flag bit 58 + flag bit 59 | binary group 8            |
| 8–9  | 64–79 | synchronization word (fixed `0xFC 0xBF`) |                             |
