# Common Encryption boxes (ISO/IEC 23001-7) — #429, #564

**Source situation:** 23001-7 is paid (no free clean full-text; W3C EME-MP4 defers to
it). `sinf`/`frma`/`schm`/`schi`, `saiz`/`saio`, `sbgp`/`sgpd` are in the **vendored**
14496-12 (`isobmff-sample-groups.md`, §8.12 Protection Scheme). `tenc`/`senc`/`pssh`
layouts below are pinned by the **byte-exact ffmpeg oracle** (`fixtures/mp4/cenc.mp4`,
see `FMP4-GAPS-ORACLE.md`) + the well-established 23001-7 structure.

**Independent corroboration added for #564** (real box-level implementation status:
`tenc`/`senc`/`saiz`/`saio`/`sinf`/`schm`/`schi`/`pssh` box parse+serialize are **already
fully implemented** — #444, #465, #480, #550 — issue #564's "no boxes" framing was stale
by the time this was checked). The `tenc` layout below was cross-checked against two
independent, actively-maintained open-source implementations that were not consulted when
this doc was first written: **Shaka Packager** (`box_definitions.cc`,
`TrackEncryption::ReadWriteInternal`) and **Bento4** (`Ap4CommonEncryption.cpp`,
`AP4_CencTrackEncryption::Parse`). Both agree on every field below byte-for-byte, with one
resolved discrepancy: Bento4 gates `default_constant_IV` presence on
`default_Per_Sample_IV_Size == 0` alone, while Shaka (and this doc, written independently
beforehand) gates it on `default_isProtected == 1 && default_Per_Sample_IV_Size == 0` —
this doc's existing condition (below) already matches Shaka's, confirmed correct by this
independent check rather than assumed.

Two new real fixtures back the **remaining** #564 gap (`cbcs` decrypt + the `Encrypt`
path, neither implemented yet — see "Remaining gap" below): `fixtures/transmux/h264_cenc.mp4`
and `h264_cbcs.mp4`, produced by Bento4's `mp4encrypt` (not ffmpeg, which doesn't cleanly
support `cbcs`) and independently verified via `mp4dump`/`mp4decrypt` — see
`fixtures/transmux/h264_cenc-PROVENANCE.md`. These add a **`cbcs` fixture** the existing
`fixtures/mp4/cenc.mp4` oracle doesn't cover at all.

## tenc — TrackEncryptionBox  (oracle body `00000000 00 00 01 08 <16-byte KID>`)
```
aligned(8) class TrackEncryptionBox extends FullBox('tenc', version, 0) {
    unsigned int(8) reserved = 0;
    if (version==0) unsigned int(8) reserved = 0;
    else { unsigned int(4) default_crypt_byte_block; unsigned int(4) default_skip_byte_block; }  // cbcs
    unsigned int(8)  default_isProtected;
    unsigned int(8)  default_Per_Sample_IV_Size;
    unsigned int(8)  default_KID[16];
    if (default_isProtected==1 && default_Per_Sample_IV_Size==0){
        unsigned int(8) default_constant_IV_size; unsigned int(8) default_constant_IV[default_constant_IV_size];
    }
}
```

## senc — SampleEncryptionBox  (oracle `00000002 <IVs+subsamples>`, flags bit1 = UseSubSampleEncryption)
```
aligned(8) class SampleEncryptionBox extends FullBox('senc', 0, flags) {
    unsigned int(32) sample_count;
    for (i=0;i<sample_count;i++){
        unsigned int(8) InitializationVector[Per_Sample_IV_Size];
        if (flags & 0x000002) {   // UseSubSampleEncryption
            unsigned int(16) subsample_count;
            for (j=0;j<subsample_count;j++){
                unsigned int(16) BytesOfClearData;
                unsigned int(32) BytesOfProtectedData;
            }
        }
    }
}
```
`saiz`/`saio` (14496-12 §8.7.8-9) carry the per-sample aux-info sizes + the offset to
the `senc` aux data. `senc` is paired with a `seig` (`sgpd` grouping_type) for
constant-per-group encryption.

## pssh — ProtectionSystemSpecificHeaderBox
```
aligned(8) class ProtectionSystemSpecificHeaderBox extends FullBox('pssh', version, 0) {
    unsigned int(8) SystemID[16];
    if (version>0){ unsigned int(32) KID_count; { unsigned int(8) KID[16]; } [KID_count]; }
    unsigned int(32) DataSize;
    unsigned int(8)  Data[DataSize];   // DRM-system-specific init data
}
```
(ffmpeg emits no `pssh` without a DRM system → cover with a spec-vector unit test.)

## Remaining gap (post-corroboration re-scope of #564)

Everything above (every box: `sinf`/`frma`/`schm`/`schi`/`tenc`/`senc`/`saiz`/`saio`/`pssh`)
is implemented and tested in `transmux::cenc` (`#444`/`#480`). `transmux::cenc_decrypt`
implements the **`cenc` (AES-CTR) decrypt path only** — its own module doc is explicit:
`cbcs` "**Unsupported** (returns `Error::InvalidInput`)". There is no `Encrypt` path at all
(only `Decrypt`) for either scheme — the crate can parse/build the box tree but cannot itself
turn cleartext samples into a `cenc`- or `cbcs`-protected file.

Two pieces of real, well-scoped, no-purchase-required work remain:

1. **`cbcs` decrypt** — AES-128-CBC with the pattern cipher (`default_crypt_byte_block`:
   `default_skip_byte_block`, typically `1:9`): decrypt `crypt_byte_block` 16-byte blocks,
   pass `skip_byte_block` 16-byte blocks through clear, repeat across the subsample's
   protected range; IV is the (usually constant, `tenc`-version-1) IV re-applied at the start
   of *each* encrypted run, not chained across the whole sample the way `cenc`'s CTR counter
   is (see "`cenc` vs `cbcs`" note above). `cbc` is already an optional workspace dependency
   (`transmux/Cargo.toml`, pulled by the `sample-aes` feature) — reuse it rather than adding
   a new crypto dependency.
2. **`Encrypt` path** (both schemes) — the reverse direction: given cleartext IR samples +
   key/IV/KID, build `sinf`/`tenc`/`senc`/`saiz`/`saio` and produce ciphertext, honoring
   NAL-length-prefix clear-header/encrypted-payload subsample splitting (reuse the existing
   NAL helper per #517, same discipline `sample_aes.rs`'s HLS Sample-AES path already uses).

Golden-interop gate for both: `mp4decrypt --key <kid>:<iv> <our-encrypted-output>.mp4 out.mp4`
must succeed and produce byte-identical clear samples (Bento4 as the independent oracle,
exercised via the real fixtures in `fixtures/transmux/h264_{cenc,cbcs}.mp4` — this crate's own
decrypt path, once `cbcs` support lands, must also reproduce byte-identical clear samples from
those same fixtures).
