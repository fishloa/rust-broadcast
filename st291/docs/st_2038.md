# SMPTE ST 2038:2021 — Carriage of Ancillary Data Packets in an MPEG-2 Transport Stream

_Source: SMPTE ST 2038:2021 §4 (Tables 1–2, PDF pp. 4–6), render-verified_

This document defines the syntax and semantics for the carriage of SMPTE ST 291-1
Ancillary (ANC) Data Packets in MPEG-2 Transport Stream PES packets (§1, p. 3). The
mechanism provides a transparent data pipe so ST 291-1 ANC data packets can be
conveyed frame-accurately alongside the corresponding video. It is **not** intended
for audio carriage, and is **not** intended to carry EDH packets (Introduction, p. 2).

Normative references relevant to the wire format (§3, p. 3):
- SMPTE ST 291-1:2011 — Ancillary Data Packet and Space Formatting
- ITU-R BT.1700 (02/2005) — line numbering for SD
- ITU-R BT.1120-9 (12/2017) — line numbering for HD (1920 × 1080)
- ISO/IEC 13818-1:2018 — MPEG-2 Systems (the gray PES-header fields below are defined here)

---

## §4.1 PMT signaling (p. 4)

The ANC data stream **shall** be identified in the PMT of the MPEG-2 Program of which
it is part, using the ISO/IEC 13818-1 syntax described below.

### §4.1.1 stream_type (p. 4)

The `stream_type` **shall** be set to **`0x06`** (PES private data).

### §4.1.2 anc_data_descriptor (p. 4)

The `anc_data_descriptor`, shown in Table 1, **shall** be used in the ES loop. The tag
value **shall** be **`0xC4`** (user-defined in ATSC, DVB and SCTE).

#### Table 1 — anc_data_descriptor syntax (p. 4)

| Syntax | No. of bits | Mnem. | Valid Range |
|--------|-------------|-------|-------------|
| `anc_data_descriptor() {` | | | |
| &nbsp;&nbsp;descriptor_tag | 8 | uimsbf | 0xC4 |
| &nbsp;&nbsp;descriptor_length | 8 | uimsbf | 0x00..0xFF |
| &nbsp;&nbsp;`for (I=0; i<N; i++) {` | | | |
| &nbsp;&nbsp;&nbsp;&nbsp;descriptor() | | | |
| &nbsp;&nbsp;`}` | | | |
| `}` | | | |

**§4.1.2.1 Semantics (p. 4):** `descriptor()` — this structure may be used to convey
additional information about the ANC data component. The use is optional and currently
undefined. Compliant receive devices **shall** ignore unrecognized descriptors.

### §4.1.3 Registration Descriptor (p. 4)

The `anc_data_descriptor_tag` is in ISO/IEC 13818-1 user private data code space. The
`anc_data_descriptor` **shall** be preceded by an ISO/IEC 13818-1 `registration_descriptor`
with `format_identifier` set to **`0x56414E43`** (`"VANC"`).

The format of the ANC data PES packet is shown in Table 2.

In Table 2 below, the **gray-shaded fields** (the PES header, from `packet_start_code_prefix`
through the final PTS `marker_bit`) are the ISO/IEC 13818-1 PES-header fields and, except

mandated by ST 2038.

Notes on the syntax as rendered:
- The per-ANC-packet inner loop counter is written `for (i=0; i<N; i++)` in the PDF
  (the same `i`/`N` form as the trailing stuffing loop), and the `user_data_word` loop
  uses `j` against `data_count`.
- The padding to a byte boundary after `checksum_word` is `while (!bytealigned)` emitting
  `'1'` bits (note: padding bit value is **`'1'`**, not `'0'`).
- The trailing `stuffing_byte` loop emits 8-bit `'1111 1111'` (0xFF) values.

## §4.2 — PES packetisation / framing rules (p. 6, normative prose)

- Each ANC data PES packet **shall** only contain ANC data from **one line**.
- All ANC data packets extracted from the same line **shall** be conveyed in the same
  ANC data PES packet.
- PES packets containing ANC **shall** be ordered according to the raster line number.
- The PTS value in the ANC data packet **shall** match the PTS value in the corresponding
  video PES packet within **± 2 ms**.
- For field pictures, PTS values for ANC data PES packets **shall** match the PTS within
  ± 2 ms for field 1 of the frame with which the ANC data is associated. For 3:2 pull-down,
  the equivalent PTS values for field 1 **shall** be inferred if that field is not coded
  (due to repeat field-frame).
- The PTS values for ANC lines that belong to the same field **shall** be identical.

## §4.2.1 Semantics (p. 6)

Except as noted below, the definition of all syntax fields **shall** be as defined in
ISO/IEC 13818-1 (the gray-shaded fields in Table 2). In the case of fixed value assignment,
the values **shall** be as defined in Table 2.

### c_not_y_channel_flag (1 bit)
For **HD**, this flag, when set to `'1'`, indicates the ANC data corresponds to the **color
difference channel**. When set to `'0'`, it indicates the ANC data corresponds to the
**luminance channel**. For **SD**, this flag **shall** be set to `'0'`.

### line_number (11 bits)
Contains the line number (as defined in ITU-R BT.1700 [SD] or ITU-R BT.1120-9 [HD]) that
corresponds to this ancillary data packet. The lines available to convey ancillary data are
defined in the applicable sample structure specification, and may be further restricted per
SMPTE RP 168.

### horizontal_offset (12 bits)
Defines the location of the ANC packet relative to SAV. `0` means that the ADF sequence
begins immediately following SAV. For **HD**, this is in units of Y samples (e.g., the range
0 to 2199 for 1080i @ 59.94). For **SD**, this is in units of (27 MHz) multiplexed word numbers.

> Note (p. 6): This field has a width of 12 bits, thus a maximum value of 4095. SMPTE ST 296
> (1280 × 720 progressive active images) image sampling systems 7 and 8 (1280 × 720 progressive
> @ 24 fps and 1280 × 720 progressive @ 23.98 fps respectively) have a luma sample number maximum
> of 4124. It is unlikely that an actual implementation would have an ANC packet begin at a
> Horizontal_Offset beyond 4092 in these formats.

### DID, SDID, data_count, user_data_word, checksum_word (each 10 bits)
These fields represent the data to be reinserted as ancillary data, per SMPTE ST 291-1. For
the purposes of the **"for" loop syntax counter only**, only the **lower 8 bits of `data_count`**,
corresponding to bits b7 (MSB) through b0 (LSB) of the 10-bit SMPTE ST 291-1 data count word,
**shall** be used.

(Per ST 291-1, these 10-bit words carry even/odd parity in the upper two bits — b9 = inverse
of b8, b8 = even parity over b0..b7 for DID/SDID/DC/UDW, and `checksum_word` is the ST 291-1
checksum. ⚠ ST 2038 itself does not restate the parity/checksum derivation in this PDF — it
defers to ST 291-1 via "reinserted as ancillary data, per SMPTE ST 291-1". The exact parity/
checksum math must be taken from ST 291-1, not from ST 2038.)

### stuffing_byte (8 bits)
A fixed 8-bit value equal to `'1111 1111'` (0xFF) that may be inserted by a transmitting
device. It may be ignored by the receiving device.

## ⚠ Flags for re-check against the PDF

1. **Table 1, loop counter casing (p. 4):** the PDF renders the for-loop init as
   `for (I=0; i<N; i++)` — a capital `I` in the initializer but lowercase `i` in the
   condition/increment. Transcribed verbatim; this is almost certainly a typo in the
   spec for lowercase `i`.

2. **Table 2, inner ANC loop counter (p. 5):** both the per-ANC-packet loop and the
   trailing stuffing loop are printed as `for (i=0; i<N; i++)` using the same `i`/`N`
   names. Verbatim from the render. (The two loops are sequential, not nested — both
   iterate "N" ANC packets / stuffing bytes respectively; `N` is not otherwise defined
   numerically in the table.)

3. **Padding bit after checksum_word (p. 5):** the `while (!bytealigned)` body emits
   `'1'` bits (not `'0'`). Confirmed from render.

4. **`data_count` width vs. loop counter (pp. 5–6):** the field is **10 bits** wide on
   the wire (Table 2), but §4.2.1 says only the **lower 8 bits** are used as the
   `user_data_word` loop counter. Both facts captured; flagging the 10-vs-8 distinction
   as the easy-to-misimplement point.

5. **Parity/checksum (p. 6):** ST 2038 does **not** define the 10-bit parity bits or the
   checksum algorithm itself — it only says the words are "reinserted as ancillary data,
   per SMPTE ST 291-1." Any parity/checksum logic must come from ST 291-1 (not vendored
   in this PDF). The parenthetical derivation I added under DID/SDID is from ST 291-1
   general knowledge, not from this PDF — verify against ST 291-1 before implementing.
