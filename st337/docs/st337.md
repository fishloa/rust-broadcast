# SMPTE ST 337:2015 — Format for Non-PCM Audio and Data in an AES3 Serial Digital Audio Interface

Curated transcription of the normative text, fetched directly from
`https://pub.smpte.org/latest/st337/st0337-2015.pdf` (17pp, freely available —
the SMPTE catalog is public at pub.smpte.org). This is the implementation/audit
oracle for this crate — cite this file, not the module doc comments, when
checking field semantics. Page numbers below are the document's own ("Page N
of 17 pages"), not PDF page indices.

Informative sections (Foreword, Introduction, Annexes A/B/C) are not
transcribed verbatim here beyond what's needed for the design notes at the end
of this file — see the PDF directly for prose color.

## 1 Scope (p.4)

> This standard specifies an interface format for the transport of non-PCM
> audio and data in professional applications using the AES3 serial digital
> audio interface. This standard includes both physical and logical
> specifications, based on the existing AES3 format, to allow exchange of
> non-PCM data between different devices.

This crate implements only the **logical burst-preamble/burst-payload framing**
(§7) — never the AES3 physical/electrical layer (biphase-mark line code,
subframe/timeslot bit placement). See "Scope decision" below.

## 4 Definitions (p.5)

Bit-field values are given in single quotes, MSB (leftmost) first, e.g. `'100'`
= decimal 4.

### 4.1 Frame Frequency

The frequency at which AES3 frames occur; equals the linear-PCM sample rate
when carrying PCM audio.

## 5 Overview (p.5)

AES3's logical format is a sequence of subframes; each subframe carries one
linear-PCM sample in 32 time slots (4 of which are sync preamble). A pair of
subframes (one per channel) makes an AES3 **frame**. 192 frames make a
**block**; the 192 channel-status bits per channel per block form the 24-byte
**channel status word**.

**Table 1 — AES3-3 subframe bit field usage for non-PCM data** (p.5):

| Bit locations | AES3-3 field | Non-PCM data usage |
|---|---|---|
| 0-3 | AES sync preamble | as per AES3-3 |
| 4-7 | Aux data field / 4 LSBs of sample word | Non-PCM data in 24-bit mode (§7.2) |
| 8-27 | Audio sample word | Non-PCM data |
| 28 | Validity (V) | as per AES3-1/AES3-3 |
| 29 | User data (U) | as per AES3-2/AES3-3 |
| 30 | Channel status (C) | bytes 0,1,2,23 defined; others undefined (§6.2) |
| 31 | Parity (P) | as per AES3-3 |

Non-PCM data streams are formed into **data bursts**: a preamble (info about
the burst) followed by a payload. Bursts are placed in the sample-word/aux-data
fields of AES3-3 subframes in **frame mode** (both channels combined, up to 48
bits/frame) or **subframe mode** (each channel independent, one linear-PCM
channel + one non-PCM channel, or two independent non-PCM channels, possible).

Bursts are tagged with a `data_stream_number` (0-7, 7 reserved for the
dedicated **time stamp** data type) so up to 7 (frame mode) or 14 (subframe
mode, 7 per channel) independent non-PCM streams can be time-multiplexed.

Bursts use 16, 20, or 24 bits of the available per-subframe space
(`data_mode`).

## 6 Interface Format (p.6-8)

### 6.1 Detailed Specifications

The logical interface format is AES3-1/-2/-3 except as noted here. Non-PCM
data occupies bit locations 4-27 of the AES3-3 subframe (some or all); unused
bit locations within a subframe, or subframes between bursts, **shall be set
to `'0'`**.

### 6.2 Channel Status Word

Only bytes 0, 1, 2, and 23 of the 24-byte channel status word are defined for
non-PCM channels (rest undefined, recommended `'0'`):

- **Byte 0**: bit0=`'1'` (professional use); bit1=`'1'` (non-audio mode);
  bits2-4=`'000'` (emphasis not indicated); bit5=frame-frequency lock status
  (per AES3-2); bits6-7=frame frequency (per AES3-2; `'00'` = not indicated,
  allowed but discouraged).
- **Byte 1**: bits0-3=`'0000'` (encoded channel mode not indicated); bits4-7=
  encoded user-bits management (per AES3-2).
- **Byte 2**: bits0-2=auxiliary sample-bit usage (per AES3-2, reflecting the
  non-PCM data word length = the greatest `data_mode` in use); bits3-5=
  non-PCM data word length (per AES3-2, `'000'` = not indicated, allowed but
  discouraged); bits6-7=`'00'` (reserved).
- **Byte 23**: bits0-7 = CRCC per AES3-2; **the default `0x0` state is not
  allowed** (a valid CRCC must be computed).

Out of scope for this crate (channel-status word is an AES3-2 physical-layer
structure, not part of the §7 burst framing this crate models — see "Scope
decision" below).

### 6.3 Sample Rate Synchronization

No requirement is placed on synchronization between the AES3 frame frequency
and the non-PCM stream's own coded sample rate (informative — companion
SMPTE ST 338 documents data-type-specific sync mechanisms, e.g. the time
stamp data type).

## 7 Data Burst Format (p.8-13)

### 7.1 General

Each data burst = `burst_preamble` + `burst_payload`, in a continuous sequence
of AES3 frames. Multiple streams are time-division multiplexed.

### 7.2 burst_preamble

#### 7.2.1 General

Occurs at the start of every burst, followed by `burst_payload`.

#### Table 6 — Preamble Words (p.9)

| Word | Contents |
|---|---|
| **Pa** | Sync word 1 = `0xF872` (16-bit mode) / `0x6F872` (20-bit) / `0x96F872` (24-bit) |
| **Pb** | Sync word 2 = `0x4E1F` (16-bit mode) / `0x54E1F` (20-bit) / `0xA54E1F` (24-bit) — the PDF's own Table 6 labels this row "Sync word 1" too (apparent spec typo); the numeric constants are transcribed verbatim and are correct |
| **Pc** | `burst_info` value (§7.2.4) |
| **Pd** | `length_code` — unsigned integer, **the number of data BITS in `burst_payload`**. In the six-word preamble, Pe+Pf count as payload bytes (see below) "to ensure compatibility with equipment that does not support six subframe operation." |
| **Pe** | `extended_data_type` (values registered in SMPTE ST 338); range `0x0000`-`0xFFFF`; in 20/24-bit modes the **higher**-significance bits are zero (not decoded by this crate — carried opaque). |
| **Pf** | Reserved. Only present in the six-word preamble form. |

There are **two forms** of `burst_preamble`:

- **Four-word** (Pa-Pd): used whenever `data_type != 31`.
- **Six-word** (Pa-Pf): used **only** when `data_type == 31` — the 4-word
  form's `data_type` field is only 5 bits, so 31 is reserved as an escape to
  the (16-bit) `extended_data_type` code space in Pe.

The 16-bit-mode Pa/Pb sync-word values above (`0xF872`/`0x4E1F`) are
**independently verified**: extracting a real E-AC-3 elementary stream from
this workspace's `fixtures/ts/dolby/eac3.ts` and wrapping it with `ffmpeg -f
spdif` (which implements the IEC 61937 consumer analog of this same
burst-preamble structure) produces exactly these two words as the burst's
first four bytes. See `docs/st337-PROVENANCE.md`.

Each preamble word occupies 16/20/24 bits of a consecutive AES3-3 subframe
(one word per subframe in the four/six-subframe forms), MSB always at time
slot 27, LSB at time slot 12/8/4 depending on `data_mode`. **This crate does
not model the 20-bit/24-bit-wide physical word layout** — see "Scope
decision" below.

#### 7.2.2 Frame mode / 7.2.3 Subframe mode (p.9-10)

Describe which AES3-3 subframe(s)/channel each preamble word physically lands
in (frame mode: alternating Ch1/Ch2 across 2 or 3 frames; subframe mode: 4 or
6 consecutive subframes of one channel). This is AES3 physical/electrical
placement, out of scope for this crate (see below) — the *logical* word
sequence Pa,Pb,Pc,Pd,[Pe,Pf] is unaffected by which placement mode is used.

#### 7.2.4 burst_info (Pc) — Table 7 (p.10)

The `burst_info` word's bit layout depends on `data_mode` (which of the 16,
20, or 24-bit-wide word the burst uses); this crate implements **only the
16-bit-mode column** (see "Scope decision"):

| Field | 16-bit-mode bits | Width | Meaning |
|---|---|---|---|
| (reserved) | — | — | n/a in 16-bit mode (only present in 20/24-bit modes) |
| `data_type` | 0-4 | 5 | unsigned, 0-31; MSB at bit4, located at AES3-3 time slot 16 (§7.2.4.2) |
| `data_mode` | 5-6 | 2 | Table 8 (below); MSB at bit6, time slot 18 |
| `error_flag` | 7 | 1 | `1`=burst may contain errors, `0`=data may be valid; time slot 19 |
| `data_type_dependent` | 8-12 | 5 | meaning depends on `data_type`, defined in data-type-specific specs (ST 338) — opaque to this crate |
| `data_stream_number` | 13-15 | 3 | MSB at bit15, time slot 27; which of up to 7/14 streams this burst belongs to |

All bit positions independently verified against the real ffmpeg-oracle burst
(`docs/st337-PROVENANCE.md`): the extracted `Pc = 0x0015` decodes as
`data_type=21` (bits0-4=`0b10101`), `data_mode=0`/16-bit (bits5-6=`00`),
`error_flag=0` (bit7=`0`) — internally self-consistent with the E-AC-3 payload
that follows.

##### 7.2.4.2 data_type

5-bit code (0-31); the type -> codec/format mapping is defined by companion
spec **SMPTE ST 338** ("Format for Non-PCM Audio and Data in AES3 — Data
Types"), which is **not fetched or verified for this crate** (it returned no
document from the same free pub.smpte.org catalog path at the time of
writing). This crate therefore does **not** define a named enum for
`data_type` values — see "Scope decision: no data_type enum" below. `31` is
reserved to mean "see `extended_data_type` (Pe)".

##### 7.2.4.3 data_mode — Table 8 (p.11)

| Value | Data mode | `burst_payload` position |
|---|---|---|
| 0 | 16-bit mode | AES3-3 subframe time slots 27-12 |
| 1 | 20-bit mode | AES3-3 subframe time slots 27-8 |
| 2 | 24-bit mode | AES3-3 subframe time slots 27-4 |
| 3 | Reserved | n/a |

May vary burst-to-burst, even within one `data_stream_number`.

##### 7.2.4.4 error_flag

See table above.

##### 7.2.4.5 data_type_dependent

5 bits, meaning defined per-`data_type` in other specs (ST 338 etc.) — this
crate carries the raw value, uninterpreted.

##### 7.2.4.6 data_stream_number

3-bit stream tag. `data_stream_number == 0` **shall** always be present in a
valid AES3 non-PCM stream and carries the main audio service. `7` is reserved
for the **time stamp** data type. `0-6` are otherwise available; up to 7
independent streams multiplexed in frame mode, 14 in subframe mode (7 per
channel, tracked independently per channel).

#### 7.2.5 length_code (Pd) (p.12)

> The length_code shall indicate the length of the burst_payload **in bits**.
> The length_code shall occupy 16, 20, or 24 bits of an AES3-3 subframe
> depending on the data mode... The burst_payload field is limited in size
> based on the data mode: from 0 to 65,535 bits in the 16-bit mode... The size
> of the burst_preamble words Pa-Pd shall not be counted in the value of
> length_code.

For the six-word preamble, Pe and Pf (32 bits) **are** counted in
`length_code` even though structurally they are part of the preamble — this
is explicit compatibility scaffolding so an implementation unaware of the
six-word form still skips the right number of bytes before the next
sync/burst.

**Judgment call (important — see PROVENANCE for the real-world discrepancy
this creates):** this literal spec text is unambiguous that `length_code`
counts **bits**, and this crate implements exactly that. A real-world
cross-check against `ffmpeg -f spdif`'s E-AC-3 burst shows `Pd` there equal
to the **byte** count of the wrapped frame (834), not `834*8` bits — this is
a known IEC 61937 (consumer format) quirk specific to how the AC-3/E-AC-3
"Burst-info" part of that *different* standard defines the field for that
data type, not a bug in this crate. See `docs/st337-PROVENANCE.md`.

### 7.3 burst_payload (p.12-13)

`burst_payload` is segmented into data words placed in a continuous sequence
of AES3-3 frames, in frame mode (both channels' subframes combined per frame)
or subframe mode (each channel independent). Considering `burst_payload` as a
serial bit stream: the first bit of the first payload data word occupies the
MSB position of the first payload-carrying subframe (time slot 27); the last
bit of that data word occupies the LSB position (per `data_mode`) of the next
subframe. Any unused bits in the last frame **shall be set to `'0'`**.

This crate treats `burst_payload` as an **opaque borrowed byte slice** — it
never interprets or transforms payload content (same "parse the container,
not the codec" discipline as `transmux`).

### 7.4 Burst Spacing (p.13)

> There shall not be a sequence of 4096 or more AES3 frames (in frame mode)
> or subframes (in subframe mode) which contain at least one data burst,
> without the beginning of at least one of the data bursts preceded by four
> AES3-3 subframes which have subframe contents in time slots 8-27 of all
> 0s.

I.e. a receiver must see the extended sync pattern (4 zero words then Pa,Pb)
at least once every 4096 frames/subframes, so an auto-detector (Annex A) can
resynchronize even if it starts mid-stream. Bursts of one stream are placed in
consecutive order; multiplexing order across streams is otherwise
unconstrained by this standard.

**Out of scope for this crate** (v0.1.0): this is a multi-burst,
stream-level constraint over a continuous AES3 transport, not a single
`Burst`'s framing. A future transport/demux layer (analogous to `mpeg-ts`'s
relationship to `dvb-si`) could enforce it; this crate models one burst at a
time.

### 7.5 Data Type Dependent Fields (p.13)

`data_type_specific`/`burst_payload` content formatting is entirely
data-type-specific and out of ST 337's scope (deferred to ST 338 and other
data-type specs) — this crate never interprets it.

## 8 Consumer Format Compatibility (Informative, p.14)

IEC 61937 is the consumer analog. For explicit interop, professional sources
restrict to 16-bit frame mode (`data_mode=0`, both AES3-3 subframes used) and
`data_stream_number=0`; some professional `data_type`s may be undefined in the
consumer format and vice versa; `data_type_dependent` and `burst_payload`
coding may differ between the professional (this spec) and consumer (IEC
61937) definitions for the same nominal codec — see the PROVENANCE doc for a
concrete instance of this (the `length_code` bits-vs-bytes discrepancy).

## Annex A — Auto-Detection of Audio/Data Mode (Informative, p.15)

A receiver can detect DATA vs PCM mode without reading channel status, by
looking for the extended sync code (4 zero words then Pa, Pb — vanishingly
unlikely in natural PCM audio) over a rolling 4096-frame window. Out of scope
for this crate (stream-level, not single-burst).

## Annex B — Equipment Compatibility Issues (Informative, p.16)

Devices may truncate 20-/24-bit data modes to 16 bits; receivers should
determine `data_mode` from the burst preamble's own field, not from channel
status. Non-PCM data may be corrupted by devices not compliant with this
standard (truncation, dithering, gain/rate conversion, etc.) — encoded data's
own integrity mechanisms (CRC, checksums, parity) are the mitigation, not
anything this standard itself provides.

## Annex C — Bibliography (Informative, p.17)

SMPTE ST 12-1:2014 (Time and Control Code); AES3-4-2009 (Physical/Electrical);
IEC 61937-1 Ed 2.1 2011-12 (General); IEC 61937-2 Ed 2.1 2011-12
(Burst-info); further IEC 61937 parts define specific data types.

## Scope decisions made by this crate (not verbatim spec text)

1. **Container layer only, never AES3 physical layer.** This crate parses/
   builds the *logical* burst-preamble + burst-payload word sequence
   (Pa,Pb,Pc,Pd,[Pe,Pf],payload) as a plain byte stream. It does **not** model
   AES3-3 subframe/timeslot bit placement, frame-vs-subframe physical mode,
   channel-status-word content (§6.2), or biphase-mark line coding — all of
   that is the AES3 *transport* this format rides over, analogous to how
   `transmux` parses container/codec-config headers but never decodes audio
   samples. Input to `Parse`/output from `Serialize` is `&[u8]`, 2 bytes per
   16-bit preamble word, matching how the spec itself tabulates Pa-Pf as
   16-bit-mode values.

2. **16-bit `data_mode` only (v0.1.0).** §7.2.1's `data_mode` field is a
   genuine wire value this crate parses/round-trips faithfully (`DataMode`
   enum: `Mode16`/`Mode20`/`Mode24`/`Reserved`). However, the 20-bit and
   24-bit modes imply *wider* (20/24-bit) physical preamble words with their
   own distinct fixed sync-word constants (Table 6) that don't fit this
   crate's uniform 2-bytes-per-word byte-stream abstraction (a 20-bit word is
   not a whole number of bytes). `BurstPreamble::parse`/`serialize_into`
   therefore accept only `data_mode == Mode16` and return
   `Error::UnsupportedDataMode` otherwise, rather than silently mis-parsing.
   Modeling 20-/24-bit-wide words is left as future work if a real need
   arises (this workspace's only real-world oracle, `ffmpeg -f spdif`, only
   ever produces 16-bit mode, matching consumer-compatible practice per §8.3).

3. **No `data_type` enum.** The `data_type` -> codec/format mapping is
   defined by SMPTE ST 338, which this crate does not have truthful,
   independently-verified access to (see the data_type section above). Per
   this project's "no implementation without a truthful source" discipline,
   `data_type` is exposed as a validated `u8` (0..=31), not an enum with
   invented codec-name variants. `DataMode` (Table 8, defined directly in
   *this* fetched spec) is the only field-enum this crate introduces, and it
   carries the `name()`/`Display` label pair per the workspace's #204
   convention.

4. **`length_code` is bits, per the literal spec text**, even though this
   creates a documented, verified discrepancy against `ffmpeg -f spdif`'s
   real E-AC-3 IEC 61937 output (which uses bytes for that data type) — see
   `docs/st337-PROVENANCE.md`. This crate implements SMPTE ST 337 (the
   professional spec), not IEC 61937.

5. **Six-word (`Pe`/`Pf`) preamble.** `BurstPreamble` carries `extended:
   Option<ExtendedPreamble>`, required to be `Some` iff `data_type == 31` and
   `None` otherwise (enforced on both parse and serialize/build). `Pf`
   (reserved) is preserved byte-for-byte on round-trip rather than forced to
   zero — the spec's own conformance-notation section (§2) defines
   "reserved" as "not defined at this time... may be defined in the future,"
   not "must be zero," and forcing zero would silently corrupt a real wire
   value from equipment using a not-yet-defined Pf convention.

6. **`§7.4` burst spacing and Annex A auto-detection** are multi-burst,
   continuous-stream concerns, out of scope for a single `Burst`'s
   parse/serialize (see inline notes above).
