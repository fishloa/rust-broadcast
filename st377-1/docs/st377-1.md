# SMPTE ST 377-1:2019 — Material Exchange Format (MXF) File Format Specification

Curated transcription for the `st377-1` crate, fetched directly from
`https://pub.smpte.org/latest/st377-1/st377-1-2019.pdf` (187 pages, free per
SMPTE's 2026-06-17 catalog release) and converted with the project's
verified `pdf2md` pipeline (`--engine textlayer`, exit code 0 — every
`0x`/`0b`/digit-run token in this transcription was diffed against the PDF's
own text layer and matched; no manual re-typing of hex/decimal values).

This file is the field-semantics oracle for this crate — cite section
numbers from here (and, transitively, from the spec itself), not from the
crate's doc comments.

## Scope decision for this crate (read first)

ST 377-1 defines the *whole* MXF ecosystem's structural skeleton, but the
bulk of its normative content (Annexes B-H: Generic Package/Track/Sequence/
SourceClip variants per essence kind, DM/Application Metadata plug-in
mechanisms, all Picture/Sound/Data Essence Descriptor property definitions,
Operational-Pattern-specific constraints) is either (a) essence-kind- or
operational-pattern-specific, meaning a "complete" typed model would need to
anticipate every OP/essence-mapping sibling spec that hasn't been
implemented yet, or (b) informative material with no wire encoding of its
own. Per issue #672 and the project's #576 opaque-fallback precedent, this
first pass draws the line as follows:

**Fully typed** (this is the format's own backbone, true regardless of
operational pattern or essence type):

- KLV framing: 16-byte Key + BER-encoded Length + Value (§6.3.4,
  ISO/IEC 8825-1 BER short/long form) — [`KlvItem`].
- The Partition Pack (§7.1, Table 5) in all three flavours — Header/Body/
  Footer (§7.2-7.4) — including the Header/Body/Footer discriminant and the
  Open/Closed × Complete/Incomplete status byte (§6.2.3, Table 4 byte 15) —
  [`PartitionPack`], [`PartitionKind`], [`PartitionStatus`].
- The Primer Pack (§9.2, Tables 13-15): the read-only local-tag → UL/UUID
  lookup table that resolves every 2-byte tag used elsewhere in the same
  Partition's Header Metadata — [`PrimerPack`].
- The "local set" KLV-lite framing used by every Header Metadata Set
  (§9.3, §6.3.5): `{local_tag: u16, length: u16-or-BER, value}` items inside
  an outer 16-byte Set Key + BER Length — [`LocalSet`]. The Set Key's byte 6
  selects 2-byte-only vs. BER item-length encoding (§9.3 Note 1); every
  Table 17 Set Kind (byte 14/15) is enumerated as [`StructuralSetKind`] so
  *any* structural metadata set can be identified even when this crate
  doesn't type its properties.
- The Random Index Pack (§12, Tables 29-30) — [`RandomIndexPack`].

**Typed** (the four Root Metadata Sets, Annex A — every real MXF file has
exactly one of each, regardless of Operational Pattern):

- Interchange Object (A.1) — the abstract superclass every Set's common
  properties are drawn from (Instance UID, optional Generation UID/Object
  Class/ApplicationPlug-In batch) — folded into each typed set below rather
  than a separate public type, since it is never instantiated on its own.
- Preface (A.2) — [`Preface`].
- Identification (A.3) — [`Identification`].
- Content Storage (A.4) — [`ContentStorage`].
- Essence Container Data (A.5) — [`EssenceContainerData`].

Each typed set is a thin decode-on-demand wrapper over its own [`LocalSet`]
(the same pattern `dvb-si`'s `DvbText` uses over raw wire bytes): parsing
never re-encodes, so a real file's exact byte layout — including any dark
(unrecognized) tags, non-canonical tag ordering, or a decoder-hostile
mixture of 2-byte/BER item lengths in the same file across different
Partitions — round-trips byte-for-byte. A typed *builder* path
(`Preface::new` etc.) constructs a fresh, canonically-ordered `LocalSet`
from typed fields for the "construct new, not just parse-preserve" gate.

Properties marked "dyn" in the spec's own tables (no fixed static local
tag — Annex H does not assign one, so a real Primer Pack must be consulted
to learn which 2-byte tag *this file* used) are resolved through
[`PrimerPack::resolve_tag`] rather than a compile-time constant; every other
property in Annex A rows below carries a fixed, spec-registered static tag
(§9.2.2 — new specs are told not to allocate new static tags, but the
Root Metadata Sets' own tags predate that deprecation and are permanent).

**Identified but generic** (everything else): any other Header Metadata Set
(Material/Source Package, Generic Track, Sequence, SourceClip, any
Descriptor, Locators, DM/Application plug-in objects, …) parses as a
[`LocalSet`] tagged with its [`StructuralSetKind`] (or
`StructuralSetKind::Unknown` for a non-Table-17 byte 14/15 pair, e.g. a
private/dark extension) — its raw tag/value pairs are preserved and
re-emitted byte-identically, but this crate does not decode individual
properties inside them. A future story can add a typed wrapper the same way
`Preface`/`Identification` do, without touching this crate's outer framing.
The Index Table Segment (§11.2.2, Table 25) is a **separate Key family**, not
part of Table 17/16's Local Set pattern: byte 11 (Application) is `0x02`
("MXF File Structure") there, versus `0x01` ("MXF/AAF Association Structural
Metadata Sets") for every Table 16/17 Structural Metadata Set — so an Index
Table Segment's Key correctly fails this crate's `is_local_set_key` check and
is never misidentified as one. Index Table Segments are entirely out of scope
for this crate (they live in the Partition's Index Table, not Header
Metadata) — an earlier draft of `StructuralSetKind` incorrectly included an
`IndexTableSegment` variant claiming byte 14/15 `0x01`/`0x10`; that variant
never matched a real on-wire Index Table Segment (whose actual Key differs at
byte 11) and has been removed rather than left half-correct.

**Out of scope entirely**: Essence Container payload bytes (the actual
audio/video/data samples carried in Body Partitions) — this crate identifies
and preserves them as an opaque `EssenceElement { key: [u8; 16], value: &[u8]
}` (via the generic [`KlvItem`] the Body Partition walker yields) but never
decodes a single codec bitstream, mirroring `st337`'s `burst_payload` and
`rdd29`'s `AudioDataDLC` boundary. Operational Patterns (§8) themselves are
represented only as the raw 16-byte UL copied verbatim from the Partition
Pack / Preface — this crate does not enumerate or validate specific OP
values (that belongs to each OP's own future sibling spec, e.g. OP-Atom).
Index Table *contents* (§11 proper — edit-unit-to-byte-offset mapping) are
not decoded, only identified (see above). UMID (`PackageId`, §4.2 "Package
ID") is carried as an opaque, fixed-size 32-byte value (SMPTE ST 330's own
internal bit layout is a separate normative reference, out of scope here,
the same way UL/UUID storage-order swapping (§4.2.1) is implemented but
UUID's own RFC 4122 internal version/variant bits are not decoded).

## §4.2 Simple Data Types (relevant subset)

- **UL**: 16-byte SMPTE Universal Label (ST 298). Stored value-order
  (`06.0E.2B.34…`, first bit always 0) — no byte swap.
- **UUID**: 16-byte RFC 4122 UUID.
- **AUID**: a 16-byte field that holds *either* a UL *or* a UUID. A UUID
  stored in an AUID field has its top and bottom 8 bytes swapped from its
  natural UUID storage order, so that the AUID's first bit is always 1 (a
  UL's first bit is always 0) — this makes the two cases distinguishable by
  inspecting bit 0 alone (§4.2.1, Table 1/Table 2). This crate exposes
  [`Auid`] with `.as_ul_bytes()` (no swap) / `.as_uuid_bytes()` (swap back)
  accessors keyed off that top bit.
- **Boolean**: 1 byte; `0x00` = FALSE, encoders emit `0x00`/`0x01`, decoders
  accept any nonzero as TRUE (§4.2).
- **UInt8/16/32/64, Int8/16/32/64**: big-endian (§6.4.2).
- **Version Type**: `UInt16`, `major*256 + minor` (§4.2).
- **Package ID**: a 32-byte Basic UMID (ST 330) or 32 zero bytes
  ("terminate a reference chain"); opaque `[u8; 32]` here (see Scope).
- **Length/Position**: `Int64`; out of scope here (no typed set below uses
  them).

## §4.3 Compound Data Types (relevant subset)

- **Batch** / **Array**: 8-byte header (`count: u32`, `item_len: u32`, both
  big-endian) followed by `count` fixed-size elements. Batch = unordered,
  no duplicates on encode (decoders must still tolerate duplicates); Array =
  ordered. Every Batch/Array this crate types (`EssenceContainers`,
  `DMSchemes`, `Identifications`, `Packages`, `EssenceContainerData`) has
  16-byte elements (UL or StrongRef/Instance-UID), so [`parse_uid_batch`] /
  [`serialize_uid_batch`] cover all of them with one helper.
- **Timestamp**: 8 bytes, big-endian fields in order: `year: Int16`,
  `month: UInt8`, `day: UInt8`, `hour: UInt8`, `minute: UInt8`,
  `second: UInt8`, `msec/4: UInt8`. All-zero = "unknown" (should be avoided,
  not forbidden) — [`MxfTimestamp`].
- **ProductVersion**: 10 bytes, 5× `UInt16` big-endian: major, minor,
  tertiary, patch, release. `release` is enumerated 0=Unknown, 1=Released,
  2=Development, 3=ReleasedWithPatches, 4=PreReleaseBeta, 5=Private (§4.3) —
  [`ProductVersion`], [`ReleaseType`].
- **String (UTF-16)**: fixed big-endian UTF-16 (BMP chars 2 bytes, others 4
  via surrogate pairs per ISO/IEC 10646 Annex Q); length is whatever the KLV
  encoding says, not null-terminated unless padding is needed. Decoded with
  `char::decode_utf16` (`core`, no_std-safe) into an owned `String`
  (`alloc`-gated) — [`decode_utf16_be`] / [`encode_utf16_be`].

## §6.3 KLV Coding

- **§6.3.3 KLV Fill Item**: key
  `06.0E.2B.34.01.01.01.02.03.01.02.10.01.00.00.00` (RP 210); MXF decoders
  must ignore byte 8 (version) when matching — some old encoders wrote
  `06.0E.2B.34.01.01.01.01.03.01.02.10.01.00.00.00` instead. Shortest legal
  Fill item is 17 bytes (16-byte key + 1-byte zero-length BER, no value).
  [`FILL_ITEM_KEY`] matches byte 8 as a wildcard per this rule.
- **§6.3.4 KLV Lengths**: encoders may use short or long-form BER, never
  exceeding a 9-byte long-form length (1 length-of-length byte + up to 8
  length bytes); the "unspecified length" token `0x80` alone (long-form
  header byte with 0 following bytes) must never appear in an MXF file.
  Decoders must handle both forms and must not assume a fixed length-field
  width.
- **§6.3.5 Local Set Lengths**: decoders must handle both 2-byte-tag/
  2-byte-length and 2-byte-tag/BER-length local sets.
- **§6.3.8**: all MXF Keys/ULs are exactly 16 bytes.
- **§6.4.2 MXF Byte Order**: everything except essence payload bytes is
  big-endian.

### BER length (ISO/IEC 8825-1, as constrained by §6.3.4)

- **Short form**: one byte, high bit clear, value 0-127 is the length.
- **Long form**: first byte's high bit set; low 7 bits = number of
  following big-endian length bytes (1-8 here, since MXF caps the *total*
  BER length encoding at 9 bytes = 1 header byte + up to 8 value bytes).
  `0x80` (long-form header claiming zero following bytes = ASN.1
  "indefinite length") is forbidden standalone in MXF.

## §7.1 Partition Pack (Table 4/5) — the file's own backbone

### Key (Table 4) — common 16 bytes, byte 14/15 vary by Partition kind/status

`06.0E.2B.34.02.05.01.vv.0D.01.02.01.01.<kind>.<status>.00`
(byte 8 `vv` = registry version the key first appeared in; this crate
writes `0x01` and accepts any value on parse per §6.3 Note; byte 13
"Structure Kind" is a fixed `0x01`, separate from byte 12 "Structure
Version" which is also `0x01`).

| Byte 14 (kind) | Byte 15 (status) |
|---|---|
| `0x02` Header (Table 6) | `0x01` Open+Incomplete |
| `0x03` Body (Table 7) | `0x02` Closed+Incomplete |
| `0x04` Footer (Table 8, closed only) | `0x03` Open+Complete |
| | `0x04` Closed+Complete |

Footer Partitions are never Open (§7.4.1 note).

### Value (Table 5), in this fixed order after Key+Length

| Field | Type | Bytes | Notes |
|---|---|---|---|
| MajorVersion | UInt16 | 2 | Fixed `0x0001` |
| MinorVersion | UInt16 | 2 | Fixed `0x0003` for this revision (`0x0002` = 377M-2004) |
| KAGSize | UInt32 | 4 | `0`=undefined, `1`=byte-aligned (no grid), `2..=1048576` valid, `>1048576` forbidden (§6.4.1) |
| ThisPartition | UInt64 | 8 | Byte offset of *this* Partition from start of Header Partition (always 0 for the Header Partition itself, regardless of Run-In) |
| PreviousPartition | UInt64 | 8 | 0 for the first Partition after the Header |
| FooterPartition | UInt64 | 8 | 0 if unknown/not yet written |
| HeaderByteCount | UInt64 | 8 | Bytes of Header Metadata + Primer Pack (+ trailing Fill) in *this* Partition; 0 if none |
| IndexByteCount | UInt64 | 8 | Bytes of Index Table Segments (+ trailing Fill) in *this* Partition; 0 if none |
| IndexSID | UInt32 | 4 | 0 = no Index Table Segment here |
| BodyOffset | UInt64 | 8 | Essence Container byte offset within its BodySID stream |
| BodySID | UInt32 | 4 | 0 = no Essence Container data here |
| OperationalPattern | UL | 16 | Copy of Preface's OperationalPattern; all-zero UL is not valid but this crate does not reject it (identification only, see Scope) |
| EssenceContainers | Batch of UL | 8+16n | May legitimately be an empty (`n=0`, 8-byte header only) Batch |

## §7.2-7.4 Header/Body/Footer Partition rules (encoded in [`PartitionPack`] + caller discipline, not separately typed structs)

- Header: `PreviousPartition` always 0; `HeaderByteCount` always nonzero
  (Header Metadata is mandatory here).
- Body: zero or more; `HeaderByteCount`/`IndexByteCount`/`BodySID` each
  independently optional (all-zero-able).
- Footer: exactly the Partition where `FooterPartition == ThisPartition`;
  `BodyOffset`/`BodySID` always 0 (no essence in the Footer); never Open.

## §9.1-9.2 Header Metadata KLV sequence + Primer Pack

Header Metadata = `Primer Pack` (mandatory first item) optionally preceded/
followed by exactly one KLV Fill item, then a sequence of Local-Set-encoded
Header Metadata Sets (first of which is always the Preface, §9.5.1), with
Fill items permitted between any of them.

### Primer Pack Key (Table 13)

`06.0E.2B.34.02.05.01.vv.0D.01.02.01.01.05.01.00` — fixed byte 14=`0x05`
("Primer Pack"), byte 15=`0x01` (Primer Pack version).

### Primer Pack Value (Table 14/15)

`LocalTagEntry` Batch: 8-byte header (`count: u32`, `item_len: u32` — always
18 for this Batch) + `count` × `{local_tag: u16, uid: AUID (16 bytes)}`. One
entry per distinct 2-byte tag used with 2-byte-tag encoding anywhere in this
Partition's Header Metadata (§9.2.1); scoped to a single Partition — never
accumulate Primer entries across Partitions (§9.2).

### Local Tag ranges (§9.2.2)

| Range | Meaning |
|---|---|
| `0x0000` | Never used |
| `0x0001`-`0x00FF` | Reserved (AAF compatibility) |
| `0x0100`-`0x7FFF` | Statically assigned (Annex H) — deprecated for *new* allocations, but the Root Metadata Sets below predate that and keep their fixed tags |
| `0x8000`-`0xFFFF` | Dynamically allocated per-Partition — must be resolved via this Partition's own Primer Pack |

## §9.3/§9.6.1 Header Metadata Set (Local Set) encoding

### Set Key common structure (Table 16)

`06.0E.2B.34.02.<xx>.01.<vv>.0D.01.01.01.01.<kind1>.<kind2>.00`

- Byte 6 `xx`: registry designator selecting the **local-set item length
  encoding** used *inside* this Set — `0x53` = 2-byte item lengths only,
  `0x13` = BER (short or long form) item lengths (§9.3 Note 1). This is a
  per-Set (i.e. per-instance, keyed off its own Set Key) choice, not a
  file-global one: two Sets of different types in the same file may use
  different encodings, and per §9.3, an encoder must use BER if *any*
  property's value exceeds 65535 bytes.
- Bytes 14/15 `kind1`/`kind2`: which Set (Table 17, reproduced below for the
  Root Metadata Sets this crate types, plus every other Table 17 entry so
  [`StructuralSetKind`] can identify — not decode — any of them).

### Local Set item framing (Figure 8)

Repeated `{local_tag: u16, length: u16 | BER, value: [u8; length]}` — no
padding between items other than an optional single trailing KLV Fill item
after the whole Set. Which length form applies is fixed by the Set Key's
byte 6 (see above) — this crate reads it once per `LocalSet::parse` and
uses it for every item in that Set.

## §12 Random Index Pack (Table 29/30)

Optional; if present must be the *last* KLV item in the file, immediately
following the Footer Partition.

- **Key** (Table 29): `06.0E.2B.34.02.05.01.01.0D.01.02.01.01.11.01.00`
  (byte 14=`0x11` Random Index Pack, byte 15=`0x01` version).
- **Value** (Table 30): `N` × `{BodySID: UInt32, ByteOffset: UInt64}` pairs
  (ascending ByteOffset order, one per Partition including Header+Footer),
  followed by a trailing `Length: UInt32` = the RIP's own total KLV length
  (Key+BER-Length+Value) — lets a decoder seek to `EOF-4`, read that
  `Length`, then seek back `Length` bytes to find the RIP's own Key, without
  parsing the whole file (§12.2 Note 2).

## Fixture provenance

No real-world `.mxf` file exists anywhere in this workspace (`fixtures/`,
`private/fixtures/`) and this is the first MXF-touching crate in the repo —
there is nothing here to extract a real capture from. `bmxlib`/libMXF-style
open-source MXF tooling repositories do ship small sample `.mxf` files, but
pulling and re-licensing a third-party binary fixture into this repo without
being able to verify its exact license terms and provenance chain first-hand
was judged riskier than the documented spec-derived fallback this project
already uses for `st12-1`/`rdd29` (`docs/CRATE-ACCEPTANCE.md` §3): every
fixture byte in `tests/fixtures/synthetic_minimal.mxf` is assembled directly
from this document's own tables (a Header Partition Pack, Primer Pack,
Preface/Identification/ContentStorage/EssenceContainerData Sets, one Body
Partition with a KLV-Fill-only placeholder "essence" element, and a Footer
Partition) by a standalone construction script independent of this crate's
own `Serialize`, then committed as a binary fixture and asserted against by
`tests/fixture_synthetic.rs` — so a bug that made `parse`/`serialize` agree
with *each other* but disagree with the spec tables above would still be
caught. If a genuinely real, clearly-licensed sample MXF file surfaces later
(e.g. via the `private/` submodule), it should be added as an additional
fixture without removing this one.
