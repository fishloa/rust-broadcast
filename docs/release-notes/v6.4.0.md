# 6.4.0 — spec-trust pass: verified docs, enum drift-guards, fidelity audit

A consolidation release focused on **trusting the spec transcriptions** and
**locking the code to them**, plus a full spec-fidelity audit. Mostly additive;
contains **one deliberate breaking change shipped as a minor** (see below).

## Spec transcriptions are now trustable (#158)

All 13 vendored ETSI/ISO spec transcriptions in `*/docs/*.md` were reconciled
against their PDFs: 8 verified-clean, 5 rebuilt from scratch (the earlier
automated rebuilds had fabricated columns, deleted/renumbered tables — those
were discarded and redone with cell-by-cell PDF confirmation behind an
adversarial verifier gate). Spec typos/inconsistencies are now reproduced
**verbatim with `> Spec note:` annotations** rather than silently "corrected",
so each doc matches its PDF *and* flags the quirk. The PDFs can now be treated
as secondary.

## Enum ↔ spec-table drift-guards (#158)

Every code-backing value→name enum (~48 across the workspace — `TableId`,
`DescriptorTag`, `StreamType`, `ServiceType`, `LinkageType`, `FecOuter`,
`Polarization`, `RunningStatus`, `PacketType`, `BufsUnit`, `SegmentationTypeId`,
`SegmentationUpidType`, `DeviceRestrictions`, …) is now mirrored in a reviewable,
spec-cited `spec_tables/*.toml` data file, with a `tests/spec_drift.rs` test that
byte-sweeps the real enum and **fails CI if the Rust enum and its TOML ever
diverge**. Zero new dependencies (an inline TOML parser keeps the MSRV-1.75 /
`--locked` lane untouched).

## Spec-fidelity audit — 2 real wire bugs fixed

An adversarial audit of every parser against the now-trusted docs found the
codebase largely spec-faithful; two genuine `dvb-bbframe` wire bugs were fixed
(both with mutation-verified regression tests):

- `Bbheader::serialize_into` in HEM / no-ISSY mode left the ISSY byte positions
  stale, corrupting a non-zero-initialised caller buffer + its CRC-8. Now
  deterministic.
- `CarryOverExtractor` mishandled `SYNCD = 65535` ("no UP starts in the DATA
  FIELD"): it discarded the carried-over partial UP instead of continuing it.
  Now handled per EN 302 755 Table 2.

Plus a sweep of misleading/incorrect doc-comments (e.g. the terrestrial
`priority` HP/LP polarity).

## Additive APIs

- **`DescriptorTag: TryFrom<u8>`** (via `num_enum`) — matches `TableId`; converts
  a known descriptor-tag byte to the typed value.
- **`SignallingKind::BufStat { bufstat, units }`** (dvb-bbframe) — decodes the
  DVB-S2 **BUFSTAT** ISSY signalling type (EN 302 307-1 Annex D Table D.1), which
  previously fell through to `Reserved`; with `bufstat_bits()`/`bufstat_bytes()`.
  EN 302 755 Annex C Table C.1 was transcribed into the T2 doc and the ISSY
  decode verified against it (#182, #183).

## ⚠️ One breaking change, shipped as a minor (deliberate)

- **`PmtSection` now exposes `section_number` / `last_section_number`** and is
  **`#[non_exhaustive]`** (#181). The PMT parser previously skipped wire bytes
  `[6]`/`[7]` and hardcoded them to `0` on serialize, so a PMT with a non-zero
  section_number did not round-trip; both fields are now parsed and preserved.
  A `PmtSection::new(..)` constructor is added for external construction.

  This is technically a breaking change (struct-literal construction and
  exhaustive field matching of `PmtSection` downstream must adapt) and would
  normally warrant a major bump. It is shipped in **6.4.0** as a deliberate
  exception: `PmtSection` is a parse-output type rarely constructed by hand, so
  real-world impact is minimal.

## Lockstep

Version-lockstep across the seven core crates (dvb-common, dvb-scte35,
dvb-bbframe, dvb-si, dvb-conformance, dvb-t2mi, dvb-tools). `dvb-stream` is
versioned/released independently and is **not** part of this set.
