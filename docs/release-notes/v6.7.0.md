# 6.7.0 — one consistent `name()` + `Display` across every public spec/field enum

A **purely additive** release (issue #204). Every public spec/field enum across
the whole workspace now exposes the same two things, so consumers never
hand-roll a `match variant => "string"` table again:

1. **`fn name(&self) -> &'static str`** — the zero-alloc static spec token;
   the reserved/unknown arm returns the static `"reserved"`.
2. **`impl Display`** — the lossless, composable view. It delegates to `name()`,
   and on a byte-bearing catch-all arm (`Reserved(u8)`, `Unallocated(u8)`,
   `UserDefined(u8)`, …) it preserves the value as `"{name}(0x{:02X})"` — e.g.
   `reserved(0x1A)`, `user defined(0x1A)`, `ISO Reserved(0x12)`.

Labels are hand-written `match` arms in source (next to each variant, greppable);
they are **not** hidden in a macro.

## `dvb-common` — `impl_spec_display!`

A new exported declarative macro `dvb_common::impl_spec_display!` generates the
mechanical `Display` impl from an enum's existing `name()`:

- `impl_spec_display!(Ty)` — `Display` is exactly `name()`.
- `impl_spec_display!(Ty, Var1, Var2, …)` — the named single-field byte-bearing
  variants render as `"{name}(0x{:02X})"`; all others delegate to `name()`.

The macro carries no labels — it only removes the otherwise-identical `Display`
boilerplate and keeps `name()`/`Display` in lockstep.

## Enforcement

Each crate gains a `label_coverage` integration test (a `src/` scan) that fails
CI if any public `pub enum` — minus a documented skip-list of errors,
dispatch/`Any*` wrappers, tag enums, section-kind discriminants, and
data-carrying ADTs — lacks a `Display` impl. Because `Display` delegates to
`name()`, this single check transitively guarantees both exist and catches any
future enum added without a label.
