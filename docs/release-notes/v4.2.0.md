# 4.2.0 — DSM-CC carousel reassembly hardening

A hardening release for the data-carousel `ModuleReassembler`, fixing two
findings from a code audit of hostile-broadcast-input handling. Behaviour is
preserved for well-formed carousels; the changes bound memory and CPU under
adversarial or corrupt input. Closes #42 and #43.

## What changed (`dvb-si`)

**Bounded slot growth (#42).** The aggregate byte budget charged only announced
`moduleSize` data bytes, so it never accounted for per-slot map and bitset
overhead — and a `moduleSize == 0` announcement cost *nothing* against the
budget while still inserting a slot. A carousel rotating
`downloadId`/`moduleId` could therefore grow the slot map without limit
(observed: 50,000 retained slots while `pending_bytes()` read 0). Now:

- `moduleSize == 0` announcements are rejected (a module with no data has
  nothing to reassemble);
- a new slot-count cap bounds the map itself — `DEFAULT_MAX_SLOTS` (16 384),
  tunable with the additive builder method `ModuleReassembler::with_max_slots`;
- `pending_bytes()` is documented as counting announced data bytes only (not a
  memory-pressure signal), and the lossy-stream starvation behaviour of the
  skip-until-space policy is documented on the type.

**O(1) stale-version lookup (#43).** `note_dii` detected stale module versions
by scanning the entire slot keyset for every announced module — O(n²) across a
DII, and a DII may announce up to 65 535 modules, making it a CPU-DoS vector
(verified quadratic: 2k→2 ms, 8k→41 ms). Slots are now keyed by
`(downloadId, moduleId)` with the version held inside the slot, so version
replacement is a single keyed lookup. Semantics are unchanged: same-version
re-announcements keep accumulated blocks, version bumps restart the module, and
mismatched-version DDBs are ignored.

## API impact

Additive only — existing constructors and the public `ModuleKey` identity on
completed `Module`s are unchanged. The one new public surface is
`ModuleReassembler::with_max_slots` and the `DEFAULT_MAX_SLOTS` constant.

`dvb-common`, `dvb-t2mi`, and `dvb-bbframe` are released in lockstep at 4.2.0
with no changes of their own.
