# v8.2.0 — 2026-07-02

Lockstep minor. Adds the **any-to-any container-mux trait vocabulary** to
`broadcast-common` (the foundation of the transmux muxing hub) and carries the
rest of the lockstep set along. Also **retires `dvb-common` from the lockstep**:
the shim is now frozen at 8.1.0.

## Highlights

- **`broadcast-common` — new `mux` module.** Four abstract, codec-agnostic traits
  that mirror the existing `Parse`/`Serialize` symmetry:
  - `Unpackage` — demux a packaged container into an in-memory media IR.
  - `Package` — mux a media IR into a packaged container.
  - `Decrypt` / `Encrypt` — in-place sample (un)protection.

  Each is generic over its input/output/media/config/key associated types plus
  `type Error`; no concrete media or codec types appear in `broadcast-common`.
  `Unpackage`⇄`Package` and `Decrypt`⇄`Encrypt` are inverse pairs. This is the
  shared hub API that every `transmux` demux/mux spoke implements.

- **`dvb-common` frozen at 8.1.0.** All workspace crates already depend on
  `broadcast-common` directly (since the 8.1.0 rename wave), so `dvb-common` is
  now a pure deprecated re-export shim with no dependents. It is dropped from the
  lockstep gate and publish list — like the other shims (`dvb-scte35`,
  `dvb-pes`, `dvb-cc`, `dvb-ule`, `dvb-smpte2038`, `dvb-emsg`). Existing
  `dvb-common = "8"` deps keep resolving (the shim re-exports `broadcast-common
  ^8`). Migrate to `broadcast-common`.

The remaining lockstep crates (`dvb-si`, `dvb-t2mi`, `dvb-bbframe`,
`dvb-conformance`, `dvb-tools`) have no source changes this release — they bump
to 8.2.0 to stay in lockstep with `broadcast-common`.

`transmux` 0.7.0 (independent cadence) ships alongside: the concrete IR
(`Media` / `Track`) + `Fmp4Demux` (`Unpackage`) and `CmafMux` / `HlsPackager`
(`Package`).
